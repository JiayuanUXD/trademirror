# TradeMirror · Claude Code 操作手册

> 项目已建好并持续迭代。本文件只保留日常开发中真正有用的模板和策略。
> 项目宪章见 `CLAUDE.md`，开发历史见 `DEVLOG.md`，产品需求见 `PRD_交易日志Web应用_TradeMirror.md`。

## 日常话术模板

### 开始新功能

```
我要实现 [功能名称]，对应 PRD 模块 [X.X]。

请先告诉我：
1. 需要改动哪些文件？
2. 数据库层需要变动吗？
3. 你打算怎么做——给我方案，不要直接写代码。
```

### 检查进度

```
请汇报当前任务的进度：
1. 已完成的部分（精确到子任务）
2. 进行中的部分
3. 遇到的问题或需要我确认的地方
```

### 代码 Review

```
请对刚才写的 [模块名] 做一次自我 Review：
1. 有哪些可以更简洁的地方？
2. 有哪些可能的 bug 或边界 case 没考虑到？
3. 有没有引入不必要的依赖或抽象？

直接说，不要美化。
```

### 架构决策讨论

```
关于 [具体问题]，我倾向于 [方案A]，但担心 [具体顾虑]。
请告诉我：
1. 我的顾虑是否成立？
2. 是否有更好的方案？
3. 给我一个明确的建议，不要给5个选项让我自己选。
```

### 调试

```
[贴上错误信息或现象]

请：
1. 解释根本原因（不是表面原因）
2. 提出修复方案
3. 修复后，说明如何避免类似问题再次发生

先解释，不要直接改代码。
```

### 结束本次对话

```
今天先到这里。请：
1. 总结今天完成了什么
2. 把今天的改动摘要追加到 DEVLOG.md
3. 提交一个 git commit（Conventional Commits 格式，message 要详细）
```

## Claude Code 使用策略

### 上下文管理
- 上下文用到一半时主动 `/clear`，长项目尤其重要
- 每次新对话开始时，让 Claude 先读 `CLAUDE.md` 和 `DEVLOG.md` 再动手
- 用 `@文件名` 引用文件，不要复制粘贴内容

### 质量控制
- 复杂任务先 `Shift+Tab` 进入计划模式，确认方案再执行
- 每完成一个子任务就 `git commit`，commit message 详细
- 每次部署前先本地跑 `pnpm tsc --noEmit` 确认无类型错误

### 失败信号（出现立刻干预）
- Claude 开始重复犯同样的错 → 上下文太长，`/clear`
- Claude 想引入新依赖解决小问题 → 拒绝，提醒 CLAUDE.md 中的"简单胜过复杂"
- Claude 想"重构所有代码" → 拒绝，让它聚焦当前任务
- Claude 不敢做决定，反复让你选 → 给更明确的边界和约束
- Claude 生成的代码自己没运行过 → 要求先跑 `pnpm build` 验证
